# CLAUDE.md — Kazooyak

A browser-based, installable, **offline** pocket chord instrument. Standalone
audio only: press pads, hear chords. Vanilla JS (no framework), bundled with
Vite. Tone.js handles audio; a pure theory module makes the musical decisions.

Kazooyak is its own project. The design was inspired by a small handheld
diatonic chord synth (see the README for context), but no code, asset, or
trademarked material is derived from it.

---

## Quick start

```bash
npm install
npm run dev        # Vite dev server with HMR
# open the URL it prints (usually http://localhost:5173)
```

```bash
npm run build      # production build into dist/
npm run preview    # serve dist/ locally to test the prod bundle + SW
```

**Deploy:** push to `main` — `.github/workflows/deploy.yml` runs `npm ci &&
npm run build` and publishes `dist/` to GitHub Pages. (One-time repo setting:
Settings → Pages → Source = "GitHub Actions".) After first load the service
worker caches everything, so the installed app runs fully offline.

**Inspect the music engine directly** — open the console and call:

```js
Kazooyak.theory.keyChords(0, 0, 'triad')   // all 7 chords in C major
Kazooyak.theory.chordNotes(7, 0, 4, '7')   // the V7 chord in G major, as MIDI notes
```

The theory module is pure ESM with no Web Audio touch, so it's directly
loadable in Node for testing:

```bash
node --input-type=module -e "import('./src/theory.js').then(t => console.log(t.keyChords(0,0,'triad').map(c=>c.label).join(' ')))"
```

## Design phase (how we got here)

The core loop: seven pads play the I–ii–iii–IV–V–vi–vii° of the current key
(Nashville Number System), a radial dial morphs chord quality in real time,
and a small control bar gates key, octave, waveform, and volume. A single-button
looper sits on top.

The decisive constraint: **iOS has no Web MIDI.** Safari and every iOS browser
(all WebKit) lack the Web MIDI API with no 2026 roadmap. So "offline iPhone
instrument" and "MIDI controller into Logic" are two different products on iOS.

Scope was then locked by these decisions:

- **Primary job:** standalone instrument (play & hear). This sidesteps the MIDI
  wall entirely — nothing in v1 needs to leave the device.
- **MVP extras beyond the core:** the **looper** only. No arp/drum/effects.
- **Sound:** "just needs to sound musical" — not a faithful emulation. So we use
  four native oscillator waveforms and a simple envelope; no wurli/piano models,
  no effects rack.
- **Quality input:** an **8-way radial dial** (chosen over a flat strip).
- **Library:** Tone.js, deliberately, so the audio plumbing is hidden and the
  *theory* module stays the legible, learnable core. (The owner wants to learn
  music theory more than Web Audio.)

## Architecture

ES modules bundled by Vite. `index.html` at the repo root loads `src/main.js`
as a module; everything else flows from there via direct imports.

```
index.html              Vite entry; loads /src/main.js
package.json            npm scripts + dependencies (tone, vite, vite-plugin-pwa)
vite.config.js          Vite config + PWA plugin (manifest + Workbox SW)
src/
  main.js               entry: imports modules, exposes window.Kazooyak, calls init()
  theory.js             pure music theory (no audio, no DOM)
  synth.js              the ONLY Web Audio boundary (Tone.js wrapper)
  looper.js             event-based looper (records notes, not audio)
  app.js                UI: builds DOM, handles touch, state, persistence
  styles.css            dark instrument theme; pads coloured by harmonic function
public/
  icons/                192 / 512 / 180 / maskable-512 PNGs (served unhashed)
.github/workflows/
  deploy.yml            CI: npm ci → vite build → GitHub Pages
```

The PWA manifest is generated by `vite-plugin-pwa` from config in
`vite.config.js` (no hand-edited `manifest.json`); the service worker is also
generated by the plugin (Workbox) and precaches the built asset shell.

### Module responsibilities

**theory.js** — the heart. Pure named exports, numbers/strings in, numbers/
strings out: `keyChords(keyOffset, octave, quality)`, `chordNotes(...)`,
`chordLabel(...)`, `chordIntervals(pad, quality)`, `midiToFreq`, `midiToName`,
`keyName`, plus constants `MAJOR_SCALE`, `DIATONIC_QUALITY`, `ROMAN`,
`QUALITY_RING`. Reference pitch: middle C = MIDI 60, A4 = 440 Hz. The eight
quality slots (`QUALITY_RING`) are `['maj/min', 7, 'maj/min7', 'maj/min9',
'sus4', 'maj6', 'dim', 'aug']`; "maj/min"-style slots stay diatonic, the `7`
slot adds a flat-7 (dominant flavour) to any triad.

**synth.js** — a thin wrapper over one `Tone.PolySynth(Tone.Synth)` into a
master `Tone.Volume`. API: `ensureStarted()` (lazy, async, calls `Tone.start()`
on the first gesture to unlock the AudioContext), `setWaveform`, `setVolume`,
`attackChord(midiNotes)`, `releaseChord(midiNotes)`, `pluckChord`, plus
`isStarted()` / `getSynth()`. Envelope is gentle (a~12ms, r~450ms) so chords
don't click. **This is the only file that touches Web Audio** — keep it that
way.

**looper.js** — records `{t, type, notes}` events relative to record start,
then replays them via a looping `Tone.Part` on the Transport. Single-button
cycle: `idle → recording → playing → idle`. Free-form loop length (no tempo/
quantize). `setOnStateChange(fn)` drives the UI.

**app.js** — exports an `init()` function called from `main.js`. Builds the
seven pads (staggered 2·4·6 / 1·3·5·7, each tinted by scale-degree colour),
the radial quality dial (top = maj/min, clockwise), the control bar (key /
octave / waveform / volume), and the looper button. Uses Pointer Events with
`touch-action: none`. Persists `{keyOffset, octave, waveform, quality}` to
`localStorage` under `kazooyak.state.v1`. Held chords store their notes in a
local closure variable so release always lets go of exactly what was pressed,
even if quality changed mid-hold.

**main.js** — composition root. Imports the modules, assigns
`window.Kazooyak = { theory, synth, looper }` for devtools, then calls
`init()`. The window assignment is **only** for console use; nothing in the
app reads from `window.Kazooyak`, so load-order is safe.

### State model

```js
{ keyOffset: 0..11,   // semitones from C
  octave:    -2..+2,
  waveform:  'sine' | 'sawtooth' | 'square' | 'triangle',
  quality:   one of QUALITY_RING (default 'maj/min') }
```

## Conventions & constraints (please preserve)

- **theory.js stays pure.** No audio, no DOM, no Tone import. It's the testable,
  console-pokable, learnable core, and the single source of truth for what's
  correct musically. Add new harmony (inversions, voicings) here first and
  verify in Node.
- **synth.js is the sole Web Audio boundary.** This is intentional: a desktop
  MIDI emitter could replace it without touching theory or UI (see deferred #6).
- **One namespace for devtools.** All modules attach to `Kazooyak` via
  `main.js`. Don't introduce new top-level globals.
- **No app-internal reliance on `window.Kazooyak`.** Imports between siblings
  are the source of truth; the window object exists for the console only.
- **Offline-first via `vite-plugin-pwa`.** Adding new top-level static assets
  generally Just Works (Workbox precaches everything matching
  `globPatterns`). If you add a file in `public/` with an unusual extension,
  update `workbox.globPatterns` in `vite.config.js`.
- **Vanilla JS, no framework.** Keep modules small and explicit. Vite is just
  a bundler + dev server — no React, no SFCs.
- **iOS realities:** the AudioContext starts suspended (handled by `Tone.start()`
  on first gesture); the viewport disables zoom to protect performance; and there
  is no Web MIDI on iOS — don't attempt to add it for the iPhone target.

## Verified

Theory output was checked against hand-worked harmony before any UI existed:
C major → C Dm Em F G Am B°; G major → G Am Bm C D Em F#dim; V7 in C → G7
(G B D F); ii with maj/min7 in C → Dm7 (D F A C); A4 = 440 Hz. If you change
`theory.js`, re-run those spot checks.

## Known issues / things to watch

- **iOS hardware mute switch** can silence Web Audio even when everything else is
  correct. Untested on a physical device. A common fix is priming a silent
  looping `<audio>` element to set the playback audio session — worth doing and
  testing on real hardware.
- **Looper** is single-layer and un-quantized; loop timing is exactly as played.
  Very long loops / dense event lists via `Tone.Part` are untested.
- `pointerleave` releases a held chord if the finger slides off the pad. Intended,
  but worth feeling out for multitouch ergonomics.

## Deferred roadmap (ranked by value / effort)

These were cut to ship the MVP. Listed roughly best-first.

1. **Press-and-hold radial quality selector.** Hold a pad → the dial pops up →
   slide to morph quality before release. Closures in `bindPad` already hold
   the notes per-press, so held chords would release correctly while the
   *next* press picks up the new quality.
2. **Inversions.** Rotate the voicing and lift the lowest note an octave for
   smooth voice-leading between chords. Cheap: a function in `theory.js` plus a
   control. Big musical payoff.
3. **Per-pad octave offsets** (independent octave per chord button).
4. **Looper overdub / layering** and optional quantize. Current looper is a
   single free-form pass.
5. **A simple reverb send** (one `Tone.Reverb`) for body.
6. **Desktop-only MIDI out** into Logic (Chrome/Edge/Firefox), gated behind
   `navigator.requestMIDIAccess` feature detection. The `synth.js` boundary
   exists precisely so this can slot in without disturbing the rest.

### Excluded from v1 entirely

Arp / Repeat / Drum / Autodrum modes; the full effects rack (reverb, delay,
flanger, tremolo, glide, LFO, filter, bass); faithful WURLI / PIANO timbres; the
randomizer; firmware/update flows. Revisit only if scope deliberately expands.
